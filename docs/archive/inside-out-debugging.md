## Problem Statement

Claude Code can't reliably work with native macOS apps. The current approach uses Peekaboo CLI, an external tool that reads the accessibility tree and takes screenshots from outside the app. It's unreliable: element IDs drift, AX trees are sparse (especially SwiftUI), accessibility permissions are fragile, and the tool itself is a point of failure in every interaction.

Meanwhile, Bolt (a native macOS email client built by the same developer) has a working "inside-out" debug infrastructure where the app instruments itself: captures its own screenshots via ScreenCaptureKit, dumps its own ViewModel state to JSON, measures its own layout via probe modifiers, and logs structured events. The agent communicates via the filesystem. No external tool, no AX tree dependency.

The problem: this proven pattern lives only in Bolt. The /browse-native skill still assumes Peekaboo. The pattern needs to be extracted into the skill so any future native app project can adopt it.

## What Makes This Cool

LLMs don't see like humans. They're pattern matchers, not eyeballs. Everyone else in this space is building "better eyes" for LLMs (MacPilot, MCP servers, accessibility APIs). This approach builds "better ears": the app tells the LLM what's happening via structured JSON. Screenshots are ground truth (what actually rendered). Structured data enables precise reasoning (exact hex colors, exact coordinates). The combination is the power.

The specific insight: browser DevTools is why Playwright works for web apps. Not just selectors, but DOM, computed styles, console output, network... all as structured text. This debug infrastructure IS the native equivalent of browser DevTools. It gives the LLM the same kind of structured data for native apps that Chrome DevTools gives for web apps.

## Constraints

- Bolt is the only native app right now. No point building abstractions for hypothetical apps.
- We don't edit Bolt code in this workspace. Bolt improvements are written as implementation plans.
- The skill should capture the pattern so CC can build custom instrumentation into future apps.
- osascript/AppleScript is the interaction mechanism (30-year standard, proven reliable).
- The skill is for the developer's own workflow, not a product for external users.

## Premises

1. **Fully abandon Peekaboo.** Inside-out replaces outside-in. The app instruments itself.
2. **Debug infrastructure is app-specific, not generic.** Each app gets custom instrumentation tailored to its ViewModels and views. The reusable part is the pattern and the skill, not a framework.
3. **osascript + filesystem triggers are sufficient for interaction.** Window management, menu access, triggering snapshots... AppleScript handles it. The app's keyboard shortcuts handle the rest.
4. **Power is in the combination of screenshots + structured data.** Screenshots are the reality check (LLM often thinks code is pixel-perfect when it isn't). Structured data enables precise comparison. Neither alone is sufficient.
5. **Scoped to Bolt today.** Future-proofing means documenting the pattern in the skill, not building abstractions. The pattern doc is aspirational until validated against a second app. That's fine. It captures intent and conventions, even if details change.

## Cross-Model Perspective

Codex independently proposed a "UI Truth Layer" concept: every action emits a before/after bundle plus a semantic diff with window z-order, focus chain, named controls, exact frames, colors (with Delta E comparisons), text, enabled/disabled state, and the last N causal events. Also proposed `bolt-debug diff latest previous` for before/after comparison and `summary.md` briefings.

Codex identified Point-Free's swift-snapshot-testing as 50% of the solution (philosophy of images + structured representations together). Missing: live in-app capture, multi-window support, app-owned probes, AppleScript triggers, LLM-first format.

Both Claude and Codex agree: don't build a generic framework. Scope to Bolt. The pattern is the product.

## Recommended Approach

**Approach B: Bolt Polish + Skill Rewrite.**

Two workstreams:


### Snapshot Bundle Spec

Every snapshot produces a directory with this structure:

```
snapshots/{timestamp}/
  manifest.json
  windows/
    main.png
    compose-1.png
    settings.png
  probes.json
  state.json
  recent-log.txt
```

**manifest.json** (required):
```json
{
  "capturedAt": "2026-04-04T13:22:35Z",
  "windowCount": 2,
  "probeCount": 24,
  "windows": [
    {
      "fileName": "main.png",
      "title": "Bolt",
      "className": "AppKitWindow",
      "frame": { "x": 0, "y": 0, "width": 1200, "height": 800 },
      "isKey": true,
      "isMain": true,
      "captured": true,
      "zOrder": 0,
      "route": "inbox"
    },
    {
      "fileName": "compose-1.png",
      "title": "New Message",
      "className": "NSWindow",
      "frame": { "x": 200, "y": 100, "width": 600, "height": 500 },
      "isKey": false,
      "isMain": false,
      "captured": true,
      "zOrder": 1,
      "route": "compose"
    }
  ]
}
```

**probes.json** (required, array of LayoutProbe):
```json
[
  {
    "id": "toolbar.archive",
    "viewType": "swiftui",
    "frame": { "x": 120, "y": 8, "width": 32, "height": 32 },
    "windowTitle": "Bolt",
    "metadata": { "enabled": "true", "label": "Archive" },
    "foregroundColor": "#007AFF",
    "backgroundColor": "#F5F5F7"
  }
]
```

New fields (Workstream 1): `foregroundColor`, `backgroundColor` as hex strings. `zOrder` and `route` in manifest window entries.

**state.json** (required): App-specific schema. Example for Bolt:
```json
{
  "capturedAt": "2026-04-04T13:22:35Z",
  "selectedFolder": { "id": 42, "name": "Inbox", "role": "inbox", "unreadCount": 12 },
  "selectedThread": { "threadId": "abc123", "subject": "Re: Design review" },
  "messageList": { "visibleMessageCount": 50, "searchQuery": null, "isSearching": false },
  "conversation": { "messageCount": 3, "messageIds": [101, 102, 103] },
  "sync": { "status": "idle", "progress": null, "outboxCount": 0 }
}
```
Schema varies per app, but should always include `capturedAt` and the current navigation/selection state.

**recent-log.txt** (optional): Last 100 lines of debug log. Plain text for humans, structured events for LLMs when available.

**Trigger protocol:** The agent writes a trigger file (`touch .context/bolt-snapshot-trigger`). The app polls for this file every 500ms via a dedicated observer (DebugSnapshotTrigger). When detected, the app deletes the trigger file, captures the full bundle, writes it to the snapshot directory, and updates the `latest` symlink (`{snapshot_base_dir}/latest` -> `{snapshot_base_dir}/{timestamp}/`). Agent waits up to 2s for `latest/manifest.json` to appear or update (poll at 200ms intervals). Total worst-case latency: ~700ms (500ms poll + 200ms capture). If the 2s timeout expires, the skill logs a warning and retries once. If the second attempt fails, it reports: "Snapshot trigger timed out. Is the app running with debug infrastructure enabled?" and falls back to `screencapture` for a basic screenshot.

### Workstream 1: Bolt Debug Infrastructure Improvements (implementation plan for Bolt repo)

These are improvements to make in Bolt's existing debug infrastructure:

1. **Multi-window manifest enrichment:** Add z-order, focus state, and route/screen identifier to each window entry in manifest.json. The agent should know which window is frontmost and what screen each shows.

2. **Color values in probes:** Add foreground and background hex color values to each LayoutProbe. Currently probes have frame + metadata but no color info. This lets the agent compare exact colors instead of guessing from pixels.

3. **osascript command catalog:** Document the osascript commands CC has discovered work with Bolt (opening compose windows, triggering menus, etc.) as a reference the skill can use.

   Core osascript primitives that replace Peekaboo:
   - Activate app: `osascript -e 'tell application "Bolt" to activate'`
   - List windows: `osascript -e 'tell application "System Events" to get name of every window of process "Bolt"'`
   - Trigger menu: `osascript -e 'tell application "System Events" to tell process "Bolt" to click menu item "New Message" of menu "File" of menu bar 1'`
   - Send keystroke: `osascript -e 'tell application "System Events" to keystroke "n" using command down'`
   - Click at coordinate: Not recommended. Use keyboard shortcuts instead (Bolt has a rich KeyboardRouter with j/k navigation, Cmd+Shift+B for snapshots, etc.)

   **What osascript CANNOT do:** Fine-grained element clicking by accessibility ID. This is intentional. The inside-out model doesn't need it. The app's keyboard shortcuts + osascript menu access cover navigation. The app's own state dumps tell the agent what happened.



4. **Snapshot summary generation:** After each snapshot, generate `summary.md` in the snapshot directory. Format:
   ```
   # Snapshot Summary
   Captured: {timestamp}
   Windows: {count} ({list of filenames})
   Focused: {window title} ({route})
   State: {1-line app state, e.g. "Inbox selected, 12 unread, viewing thread 'Re: Design review'"}
   Probes: {count} registered
   Notable: {any warnings, e.g. "Compose window open but not focused"}
   ```
   This prevents the "I only looked at main.png" problem. The agent reads summary.md FIRST, then decides which screenshots and data files to examine.

### Workstream 2: Skill Rewrite (this repo)

1. **Remove all Peekaboo references** from SKILL.md, scripts/validate.sh, scripts/detect-host-app.sh, docs/designs/native-interaction-runtime.md, README.md.

2. **Rewrite /browse-native skill** around the inside-out pattern:
   - **Setup phase:** Verify app has debug infrastructure (check for snapshot directory, trigger file mechanism). If the app lacks instrumentation, the skill operates in **degraded mode**: osascript + screencapture for screenshots, no probes, no state dumps. The skill notes what's missing and recommends adding instrumentation. For Bolt specifically, full instrumentation already exists.
   - **Interaction pattern:** osascript for window management and navigation. Filesystem trigger for snapshots. Read manifest.json to know which windows exist. Read ALL window screenshots (not just main). Read probes.json for exact measurements. Read state.json for app state.
   - **See-Act-See loop:** Trigger snapshot, read manifest + summary, read relevant screenshots + probes + state, make decision, act via osascript/keyboard, trigger another snapshot, compare.
   - **Color/alignment checking:** Always reference probes for exact values. Never rely solely on screenshot visual inspection for color or alignment claims.

3. **Update validation gates:**
   - Gate 1: Debug infrastructure produces valid snapshot bundle (manifest.json, windows/*.png, probes.json, state.json)
   - Gate 2: osascript can open windows, trigger menus, and the app responds with correct snapshots
   - Gate 3: See-act-see cycle latency <3000ms (trigger to snapshot ready to agent reads). This is relaxed from the Peekaboo-era 2000ms target to account for polling overhead.

**Workstream dependency:** Workstream 2 can start immediately. The skill rewrite targets Bolt's CURRENT infrastructure (which already has manifest.json, probes.json, state.json, screenshots). Workstream 1 enrichments (z-order, color hex, summary) are additive. The skill should handle both current and enriched formats gracefully.

4. **Document the pattern** in docs/designs/ so future apps can follow it. The pattern doc covers: what to build (DebugSnapshotService, InspectableModifier, state dumps, event logging), how the agent communicates (filesystem triggers), and what the skill expects in the snapshot bundle.

## Resolved Decisions

1. **Trigger latency:** 500ms polling is acceptable. Worst-case see-act-see cycle is ~2s (500ms poll + 200ms capture + agent reads + osascript action + 500ms poll + 200ms capture). Current Gate 3 target was <2000ms with Peekaboo. New target: <3000ms for full cycle, which is fine for debugging workflows (not real-time). FSEvents upgrade deferred to Approach C.
2. **osascript scope:** Core primitives defined above (activate, menu, keystroke). No coordinate clicking. Keyboard shortcuts are the primary interaction mechanism. osascript catalog will be validated as a prerequisite spike before full skill rewrite.
3. **Degraded mode:** YES, in scope. The skill works at three tiers:
   - **Full instrumentation** (Bolt): snapshots + probes + state + events. Best experience.
   - **Partial instrumentation** (new app, early): snapshots + state dumps, no probes yet. Good enough.
   - **No instrumentation** (any app): osascript + `screencapture` for screenshots. Window IDs obtained via `osascript -e 'tell application "System Events" to get id of every window of process "AppName"'`, then `screencapture -l {windowID} output.png`. No structured data. Functional but limited.
4. **Build integration:** The skill assumes the app is running. Building is handled by conductor-run.sh or the developer. The skill does not build.
