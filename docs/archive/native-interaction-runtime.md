# CEO Plan: Native Interaction Runtime for macOS Apps

Generated by /plan-ceo-review on 2026-03-24
Branch: kbitz/office-hours | Mode: SCOPE EXPANSION
Repo: kbitz/gstack-native

## Vision

### 10x Check
A complete native app development companion — not just QA and design review, but an AI agent that can build, launch, see, interact, find bugs, fix code, rebuild, and re-verify native macOS apps autonomously. The Playwright-grade perception-and-control layer for SwiftUI/AppKit that makes the entire gstack skill suite work for native apps.

### Platonic Ideal
The user says `/qa-native` and the agent: builds from source, launches the app, systematically explores every screen and interaction state, captures annotated screenshots with element maps, finds visual and interaction bugs, checks accessibility compliance, tests keyboard navigation, verifies dark mode and window resizing, fixes each bug in Swift source code, rebuilds, and re-verifies — all autonomously. The user feels like they have a senior QA engineer + designer + accessibility expert working alongside them 24/7.

## Accepted Scope (added to this plan)

Core (from design doc):
- Stateless Bun adapter wrapping Peekaboo CLI
- @e ref system (depth limit 8, cap 200, expandable groups)
- /browse-native skill (see, click, type, press, scroll, window, menu)
- /design-review-native skill (native-first visual QA)
- Validation against user's SwiftUI app + TextEdit (baseline)

Expansions:
- Auto-build-and-launch via xcodebuild (CLAUDE.md config: scheme, bundle ID, workspace path, configuration, launch args)
- Dark mode toggle + comparison screenshots (with appearance restore after testing)
- Accessibility report: missing labels + role coverage (contrast + VoiceOver order deferred)
- Window resize testing (predefined sizes + min/max)
- Keyboard navigation audit (tab-through with focus order report)
- Screenshot gallery (session-scoped, organized by state)
- Crash detection + repro capture (session-scoped action log, crash log correlation)

Architecture decisions (from CEO review):
- Bun.spawn() with argument arrays (no shell interpolation) — security mandate
- Session ID (not PID) for file organization — enables crash detection across commands
- Build timeout 120s default (configurable) — Codex critique accepted
- 5s timeout on all other commands
- Result type: { ok: true, data } | { ok: false, error }
- 8 error paths fully rescued (zero silent failures)
- Primary target: SwiftUI apps. Other frameworks (AppKit, Electron, Catalyst) are best-effort.

Validation gates (implemented in `scripts/validate.sh`):
- Gate 1 — AX tree quality: Peekaboo see output must contain actionable elements (buttons + text fields)
- Gate 2 — Interaction reliability: Element-ID targeting (click by AX ID, not coordinates), type with `--window-id`, state verification (typed text confirmed in AX tree), button click by ID, hotkey
- Gate 3 — Latency: see-act-see cycle <2000ms (relaxed per eng review; 1000ms triggers daemon graduation)

## Implementation Details (added after spec review)

### Session ID Convention
All files use `/tmp/native-browse-{session-id}/` where session-id is a timestamp
(e.g., `20260324-143022`). This supersedes the design doc's `{pid}` convention.
The skill markdown generates the session ID at the start of a run.

### 20-Step Reliability Proof (deferred to TODOS)
Moved to TODOS P1. The 3-gate validation proves the core loop works (AX tree
quality, element-ID interaction with state verification, latency). The 20-step
proof is better suited as a confidence test once a real SwiftUI app is the target,
since it requires app-specific navigation flows (menus, screens, save/confirm).

Definition (for when implemented):
One "step" = one see-act-see cycle (3 Peekaboo invocations).
Must complete 5/5 without losing element, window, or state.

### Timeout Policy
- `see`: 10s (complex AX trees may be slow)
- `click`, `type`, `press`, `scroll`: 5s
- `menu`, `window`: 5s
- `build`: 120s default, configurable via CLAUDE.md `native_build_timeout`
- All timeouts configurable per-command in CLAUDE.md

### Error Paths (all 8)
1. PeekabooNotFound → install instructions
2. PermissionDenied → System Settings path
3. AppNotFound → "App not running"
4. PeekabooTimeout → kill process, report
5. EmptyAXTree → "Add .accessibilityIdentifier()"
6. JSONParseError → print raw Peekaboo output
7. ElementNotInteractable → "May be disabled"
8. BuildFailed → print xcodebuild stderr

### Build Config (CLAUDE.md)
Required: `native_app_scheme`, `native_app_bundle_id`
Optional: `native_workspace_path` (auto-detect .xcworkspace),
  `native_build_configuration` (default: Debug),
  `native_launch_args`, `native_build_timeout` (default: 120)
If required fields missing: AskUserQuestion + persist to CLAUDE.md.

### Dark Mode Toggle
1. Read current appearance: `defaults read -g AppleInterfaceStyle`
2. Toggle: `defaults write -g AppleInterfaceStyle -string Dark` (or delete for Light)
3. Note: This is a SYSTEM-WIDE change. Add user confirmation before toggling.
4. Restore: After testing, restore original appearance setting.
5. Limitation: Some apps cache appearance and may need restart.

### Keyboard Nav Audit (scoped)
Scope: "Can tab through all focusable elements without getting stuck" (liveness check).
NOT: "Is the tab order correct" (requires ground truth spec).
Detects: focus traps, unreachable elements, missing focus rings.

### Crash Log Correlation
Path: `~/Library/Logs/DiagnosticReports/`
Mechanism: After detecting process exit (non-zero or signal), poll for .ips files
matching the app's bundle ID with timestamps within 10s of the crash. Max 5s wait.
Known limitation: Crash reports may be delayed beyond 5s.

### Expansion Feature Failure Modes
- build signing error → print Xcode signing error, suggest manual fix
- dark mode toggle fails → "Could not toggle appearance. App may use custom theme."
- crash log not found → "Crash detected but log not yet available. Check DiagnosticReports manually."
- resize below minimum → "Window constrained to {actual_size}. Minimum: {min_size}."
- keyboard nav: no focusable elements → "No focusable elements found. App may not support keyboard navigation."
- accessibility: no issues found → "All elements have labels and roles."
- gallery: disk full → "Screenshot gallery failed: disk space."

### Depth Limit Clarification
Depth limit 8 and cap 200 are NEW additions from the CEO review.
The original design doc did not specify these numbers.

## Eng Review Decisions (2026-03-24)

The following decisions were made during the /plan-eng-review session:

1. **Skill-markdown-first** — No Bun adapter. Peekaboo CLI invoked directly via Bash.
   Adapter deferred unless Peekaboo's native interface proves insufficient.
2. **Peekaboo native IDs** — B1, T2 etc. used directly. No custom @e ref system.
   Peekaboo's snapshot-based IDs are sufficient for the see-act-see pattern.
3. **Bundle-ID + window-ID targeting** — Deterministic app identity via
   `--window-id` (from `peekaboo window list`), not ambiguous `--app X`.
4. **Dark mode save-and-restore** — Save current appearance to session file,
   print restore command before toggling. Crash recovery via printed command.
5. **Simple build path** — `xcodebuild` happy path only. Claude troubleshoots
   build errors naturally from stderr. No error categorization code.
6. **2000ms latency target** — Relaxed from 1000ms for stateless prototype.
   1000ms becomes daemon graduation trigger (TODOS P1).
7. **Dual validation** — TextEdit baseline + user's real SwiftUI app.
8. **Permissions onboarding** — Detect and guide via `peekaboo permissions --json`.
9. **Output management** — Window-scoped via `--window-id`. Large AX trees
   documented as known limitation.
10. **Single skill** — `/browse-native` covers both interaction and visual QA.
    Split into separate skills only if markdown exceeds ~500 lines.
