## Problem Statement

`/roadmap` accreted 12+ patches across v0.8.4 â†’ v0.16.2, each fixing a specific symptom. The skill is now 1390 lines, the audit is 2726 lines. Total ~4100 lines. Several recurring pain points remain after all that patching:

1. **Tracks are too big.** plan-eng-review and plan-ceo-review routinely recommend splitting a track into 2-3 parallel tracks at session-launch time. Structure decisions get pressure-tested *after* the user has already mentally committed.
2. **Sequential vs parallel decisions are wrong.** Reviews discover mid-work that tracks need to be sequential after the user already started them in parallel â€” or that work in a future Group could parallelize with the current Group.
3. **Subcommands force the user to know which mode applies.** `/roadmap` vs `/roadmap update` requires the user to predict whether the inbox needs draining, the freshness scan needs to run, etc. The skill knows more about state than the user does.
4. **Scope creep into Future.** Items that should run in parallel with current work get parked in a future Group because the upfront analysis didn't surface the parallelizability.

Current /roadmap solves the *static plan* problem well. It does not solve the *dynamic re-planning* problem, which is now the dominant cost.

## What Makes This Cool

The skill becomes a **state machine** instead of a pipeline. One entry point. It reads what's in the repo, what changed since last run, and what the user said in their prompt, then picks the right combination of operations from a finite set: triage / closure / freshness / mid-flight revision / greenfield. No subcommands, no mode-detection ceremony, no wasted ceremony when the state is simple.

The bigger shift: **scrutiny moves earlier**. Today, /roadmap produces structure that passes its own audit but fails plan-eng-review and plan-ceo-review at session-launch time. The revamp embeds sizing + parallelism rigor *into* structure-building, so reviews then catch *taste* issues, not *shape* issues.

## Constraints

- Single entry point: `/roadmap`. No subcommands, ever.
- `/roadmap` may ask one targeted clarification question if the state is genuinely ambiguous (e.g., "you have 5 unprocessed items AND closure work, which first?"), but never demands a mode flag.
- Doc-only scope. Never modifies code.
- Total runtime budget: <5min for typical re-incorporation runs, <15min for full overhaul.
- Skill file should end up appreciably shorter than the current 1390 lines (the simplification is part of the goal), but no hard line-count target. Optimize for what works well, not for hitting a number. If the result is 1100 lines that flow cleanly, that's better than 800 lines of awkward density. Audit script gains 2 new checks + 1 modification; estimated +160 lines net, no parallel cleanup pass required.
- Parallelism cap: 4 parallel sessions max in a project (3 typical). The skill enforces this when proposing structure.
- Track size cap: roughly ~6h human / ~30min CC per track. Larger â†’ suggest split.
- Completed Group/Track names + IDs are immutable. Future-section tracks may be renamed/renumbered freely.
- Fixed vocabulary: Groups > Tracks > Tasks. No drift permitted.
- User intent embedded in prompts is honored. "Fully close out Group 2, don't defer everything" tells the skill to bias toward closure.

## Premises

1. **Single entry point with state-aware dispatch.** `/roadmap` infers what to do from current state + last-run state + user prompt. May ask one clarifier when genuinely ambiguous.
2. **Tracks are revisable hypotheses, not commitments.** Split / merge / reorder / defer / mark-parallelizable are first-class operations. Stable IDs for completed Groups/Tracks; flexibility for Future.
3. **Move scrutiny earlier.** /roadmap embeds sizing + parallelism rigor at structure time. plan-eng-review then validates rather than re-architects.
4. **Re-incorporation is the hot path.** Optimize for "I shipped 3 tracks, here are 5 new TODOs", surgical, minutes not hours.
5. **Parallelism is computed from declared metadata.** Tracks carry `touches:` (file-path globs) and `depends_on:` (track IDs) in the existing italic metadata line. `parallelizable_with:` is derived at audit time. Cap at 4. `blocks:` is *not* introduced in v1, the inverse of `depends_on:` is computed.
6. **Doc-only scope.** ROADMAP.md / TODOS.md / PROGRESS.md only. No code, no configs.

- Effort: M (human: ~3 days / CC: ~2 hours)

- Risk: Medium, loses long-range visibility, requires new skill split
- Completeness: 7/10

## Recommended Approach

**Approach A: Smart Dispatcher**

### Track Metadata Format

The current ROADMAP.md format already has separate italic metadata lines under each `### Track` heading. The format and the parser exist â€” `bin/roadmap-audit` already populates `_TRACK_TOUCHES` and `_TRACK_DEPENDENCIES` via `_parse_roadmap`. **No format change needed.** Confirming the existing convention:

```
### Track 1A: Update-Run Dir Propagation
_1 task . ~30 min (human) / ~15 min (CC) . low risk . [bin/update-run]_
_touches: bin/update-run_
_Depends on: Pre-flight 1 (requires `--skills-dir` flag to exist)._

[Optional 1-line description]

- **Task title** â€” desc. _path/to/file.ts, ~N lines._ (S/M/L)
```

- Main metadata line: tasks count, effort, risk, file list in brackets (already required).
- `_touches: <comma-sep paths>_` on its own italic line (already parsed). Audit treats tracks lacking this as `legacy` and skips them in parallelism analysis (graceful skip, no error).
- `_Depends on: <Track ID list or free text>_` on its own italic line (already parsed into `_TRACK_DEPENDENCIES`). **Contract for v1:** a Track participating in dependency logic (PARALLELIZABLE_FUTURE, dep-cycle detection) MUST use a parseable Track ID list (e.g., `_Depends on: 2C, 2D_`). Free-text values like `_Depends on: at least one major version bump..._` are valid but exclude the track from dependency analysis (treated as `legacy` for that purpose, like missing `_touches:_`). This avoids the audit hand-waving about whether free text counts as a real dep.
- Per-task `(S/M/L)` already exists. **NOT used as the size signal** â€” the existing `check_size_caps()` uses `max_tasks_per_track`, `max_loc_per_track`, `max_files_per_track` ceilings (read via `ceiling` helper). The revamp keeps that approach for consistency. The S/M/L tags stay as human-readable hints, no new role.

### Last-Run Tracking

`.context/roadmap/last-run.json` (gitignored at repo root via existing `.context/` convention). Schema:

```json
{
  "ts": "2026-04-27T13:45:00Z",
  "commit": "daa577d",
  "branch": "kbitz/roadmap-revamp",
  "state_chain": ["TRIAGE", "FRESHNESS"],
  "completed_at": "2026-04-27T13:46:12Z",
  "items_drained": 5,
  "tracks_touched": ["2A", "2B"]
}
```

The dispatcher reads this on startup. **Trust rules** (Codex caught these as edge cases):
- If absent â†’ treat as fresh start, no diff signal.
- If `branch` field doesn't match current branch â†’ treat as fresh start (the user switched branches or this is a new Conductor workspace).
- If `completed_at` is missing â†’ previous run was interrupted. Notify user, offer to resume or start fresh.
- The `.context/` directory is per-worktree, so multiple worktrees naturally have separate last-run state.

`git log --since="$LAST_RUN_TS" --pretty=format:"%H %s"` produces activity since last run. If a track was completed via squash merge or a commit message that doesn't mention the Track ID, the FRESHNESS scan will miss it â€” that's fine, the audit's STALENESS check is the authoritative signal (commit-message scanning is a hint, never a sole trigger for cleanup).

### State Decision Tree

`/roadmap` reads:
- ROADMAP.md (presence, structure, completeness)
- TODOS.md `## Unprocessed` count
- Audit output (closure debt, staleness, in-flight Groups, structural drift)
- Git log since last `/roadmap` run (commit messages mentioning Tracks)
- User prompt (closure intent, defer intent, specific Track names, "split this" / "merge these")

From these signals, it picks one or more of these operations to run:

| State | Triggered when | Operations |
|---|---|---|
| **GREENFIELD** | No ROADMAP.md exists | Build structure from scratch (current overhaul mode) |
| **DRAINED** | ROADMAP.md healthy, Unprocessed empty, no closure debt, no staleness | Exit cleanly with "Roadmap looks good" |
| **TRIAGE** | Unprocessed has items | Scrutiny gate + phase assignment + slot into existing Groups |
| **CLOSURE** | In-flight Group has 0 open Tracks OR user prompt indicates close-out intent | Mark Group âś“ Complete, prompt for hotfix vs Future on origin-tagged dangling items |
| **FRESHNESS** | Audit detects completed/stale items in ROADMAP.md | Mark complete in place, clean stale references |
| **REVISE** | User prompt mentions split / merge / reorder / parallelize specific Tracks | Apply the requested revision operation surgically |

States compose. **Fixed precedence when multiple fire:** REVISE (user explicit intent) â†’ FRESHNESS (cleanup before scrutiny) â†’ CLOSURE (close out before adding) â†’ TRIAGE â†’ DRAINED. GREENFIELD is exclusive (no ROADMAP.md = nothing else applies).

**Ambiguity handling:** Only one human prompt per run, fired only when CLOSURE and TRIAGE both have material work AND user prompt has no closure/anti-defer signal. Question: "Group N has X open-origin items (closure) AND Y unprocessed TODOs (triage). Which first?" Default: closure first.

**Edge cases:**
- ROADMAP.md exists but malformed (audit STRUCTURE fails badly): treat as GREENFIELD with prompt to confirm full restructure vs. abort.
- Mid-revision interrupt (last-run.json shows previous state chain didn't terminate cleanly): notify user, offer to resume or restart.
- No last-run.json + ROADMAP.md exists: treat as fresh start, run full FRESHNESS scan to establish baseline.

### Decision Tree (concrete)

The "state machine" framing is conceptual. The implementation is a deterministic classifier inside `bin/roadmap-audit` (which already reads everything needed) + a thin dispatcher in skill prose.

**`bin/roadmap-audit --classify-state --prompt "..."`** (new flag on existing audit, ~30 lines): consumes the audit's own findings + last-run.json + user prompt, emits JSON of triggered states. Unit-testable in `scripts/test-roadmap-audit.sh`. Single source of truth for state â€” no drift between separate audit/classifier/skill layers.

```json
{
  "exclusive_state": null | "GREENFIELD",
  "ops": ["REVISE", "FRESHNESS", "CLOSURE", "TRIAGE"],
  "needs_clarification": false,
  "clarification": null | {"question": "...", "default": "closure"}
}
```

Classifier logic (pseudocode, implemented in bash):

```
if !ROADMAP.md: emit {"exclusive_state": "GREENFIELD"}; exit
ops = []
if regex_match(prompt, revise_intent_words) and not negated: ops += REVISE
if audit.STALENESS or git_log_since_last_run shows completed tracks: ops += FRESHNESS
if audit.in_flight_groups has zero-open OR regex_match(prompt, closure_intent_words): ops += CLOSURE
if audit.UNPROCESSED.count > 0: ops += TRIAGE
if ops empty: emit {"ops": []}  (skill interprets as DRAINED)
if CLOSURE in ops and TRIAGE in ops and no prompt signal:
  needs_clarification = true
emit ops in fixed precedence order: REVISE, FRESHNESS, CLOSURE, TRIAGE
```

Skill prose calls `bin/roadmap-audit --classify-state`, parses JSON, runs the corresponding sections in order. ~30 lines of skill prose for the dispatcher, ~30 lines added to roadmap-audit + ~10 fixture tests. **JSON output: hand-rolled in bash (no jq dependency).** Schema kept flat (string fields, simple arrays) so quoting stays trivial; tests assert the schema with grep, not jq.

That's the whole "state machine". No ceremony, no drift across multiple state-reading layers.

### First-Class Revision Operations (v1 set)

Three operations in v1. Add the others if real pain emerges.

- **split-track** â€” Track too big or has internal parallelism. Operation: split into N child tracks, distribute tasks by file ownership, propagate `touches:` / `depends_on:` metadata. **In-flight tracks may be split** (premise 2: Tracks are revisable). Completed tracks may not. Original ID is retired by appending suffix `(split into 2A.1, 2A.2)` in the same line; new IDs use the parent's letter with `.1`, `.2` suffixes to preserve sort order. The retirement note stays for one /roadmap run, then is cleaned in the next FRESHNESS scan.
- **defer-task** â€” Single task within a track moves to Future. Operation: detach task, add to Future as standalone or extend an existing Future track if the user picks one.
- **surface-parallelizable** â€” Audit's PARALLELIZABLE_FUTURE check found a Future-section track whose touches don't overlap any in-flight track and whose depends_on are satisfied. Skill prose surfaces all qualifying candidates (not capped at 1 â€” the user has parallelism budget headroom and dynamic re-planning is the goal), ranked by: (1) zero file overlap with all in-flight, (2) smallest size, (3) oldest age in Future. User picks which to promote; skill performs the promote (line move from Future to current Group). **Not implemented in `bin/roadmap-revise`** â€” the surfacing IS the operation; the actual line-move on user approval reuses the existing /roadmap structural-edit path.

**Deferred to v2 unless real demand emerges:** `merge-tracks` (no observed case yet), `promote-to-current` (the user can do this manually with a one-line edit, then run /roadmap to validate), `reorder-sequential` (covered by editing `depends_on:` directly).

These are operations the *skill* invokes, not subcommands the user types. The user describes intent ("split Track 2A into auth and rendering"), the skill picks the operation.

### Upfront Scrutiny (Embedded in Structure-Building)

When /roadmap creates or revises a Track in any state, it runs three checks before finalizing:

1. **Size cap.** Estimate aggregate task effort. If >~6h human / >~30min CC, propose a split. Heuristic for the split: cluster tasks by primary file path, propose one child track per cluster.
2. **Parallelism budget.** Count current in-flight tracks + the proposed new track. If >4, surface the conflict: "You'd have 5 parallel tracks. Defer one of: ..." User picks one to defer.
3. **Dependency check.** Look at `touches:` and `depends_on:`. Flag any new track that touches files modified by an unfinished prerequisite. Either add `depends_on:` to mark sequential, or surface as "this should be sequential, confirm?"

These checks are what plan-eng-review currently catches *after the fact*. Moving them earlier means reviews validate, not re-architect.

### User-Prompt Intent Parsing

User often invokes /roadmap with a hint embedded in the prompt: "let's close out Group 2 fully, don't defer everything." Implementation is a **deterministic regex pass** over the user prompt, not an LLM classification call. Why deterministic: avoids one more LLM hop and the failure modes that drove the 12 prior patches (LLM occasionally over-/under-detects intent).

Wordlists (case-insensitive, whole-word match):

| Signal | Trigger words |
|---|---|
| `closure_intent` | close out, close-out, finish, wrap up, fully done, fully close |
| `anti_defer` | don't defer, no future, ship now, ship it all, no more future |
| `split_intent` | split, break up, break apart, too big, decompose |
| `track_ref` | regex `Track \d+[A-Z](\.\d+)?` |

If `split_intent` AND a `track_ref` is present in the same prompt â†’ REVISE state with split-track operation pre-targeted. If `split_intent` alone â†’ REVISE state, ask which track.

**Negation guard:** before firing an intent, scan a 5-word window before each trigger for negators (`don't`, `do not`, `not`, `never`, `unless`, `instead of`). If a negator appears in the window, suppress the intent. Example: "we should not split this" â†’ split_intent suppressed. False positives still possible on convoluted phrasing â€” the visible "Detected" line below is the user's correction surface.

When any intent fires, the skill prints one line: "Detected [signal] in your prompt, [behavior change]." User can override by responding "ignore that" before the next prompt â€” skill drops the bias and proceeds neutrally.
