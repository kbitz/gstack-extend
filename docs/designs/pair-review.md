# Design: /pair-review — Pair Testing Session Manager

Mode: Builder

## Problem Statement

The developer workflow goes: /office-hours → /plan-ceo-review → /plan-eng-review → build → /review → deploy → "give me a list of things to test." That last step is the weak link. Currently the test list is copy-pasted into Conductor's @notes and tracked manually. Four things break:

1. **Context loss.** The test list lives in the AI context window. After 45 minutes of fix cycles with context compaction, the original test numbers are meaningless. The agent has to regenerate the list, and continuity is lost.

2. **No checkpoint discipline.** Fixes happen inline. If a fix goes down a wrong path, reverting is painful because there's no save point before the fix attempt.

3. **Progress is mental.** What's been tested? What passed? What failed? What was added during the session? All tracked in the developer's head.

4. **Scope creep is invisible.** Fixes introduce new things to test, but the original list doesn't grow. Items get forgotten.

The developer works across multiple machines, and Conductor workspaces are ephemeral. Test state needs to survive workspace archival.

## What Makes This Cool

Every AI testing tool is trying to remove the human. This one keeps the human in the loop and gives the AI the project management role. The human provides judgment (does this feel right? is this animation smooth? does this interaction make sense?). The AI provides structure, memory, and the rebuild/redeploy loop.

The "resume" command is the product. You're 45 minutes into a test session on your laptop. You need to leave. You type `/pair-review resume` on your desktop an hour later, and the skill tells you: "You were testing auth and onboarding. 5 of 7 items in auth passed. Item 3 failed (login loading state). You fixed it in commit abc1234, which invalidated 2 related items. Rebuild command: `make dev`. Next: retest items 3, 6, 7."

That's the whoa.

## Constraints

- Must follow gstack skill conventions (SKILL.md format, preamble, telemetry, learnings, AskUserQuestion format) so it could be submitted as a PR to gstack.
- Must work for any project type: web apps, native apps, CLI tools. Not native-app-specific.
- Must handle rebuild/redeploy as part of the test-fix-retest loop.
- Architecture should make multi-agent orchestration (Approach C) a natural extension, not a rewrite. Test state format should be readable by other agents.
- State must survive context compaction, workspace destruction, and machine changes.

## Premises

1. **Some testing genuinely requires human judgment.** Subjective feel, animation smoothness, real-world usability, cross-device behavior... these can't be automated by /qa or /browse-native.
2. **The test-fix-retest loop is the core, not just checklist generation.** The skill must handle rebuilds/redeploys as part of the loop.
3. **Test state persists file-based in `.context/pair-review/`.** Self-contained, conductor-visible, no external dependencies. Cross-machine resume via conductor workspace management or committing state to the repo.
4. **General-purpose gstack skill.** Works for web apps, native apps, CLI tools... anything with a dev deployment process.
5. **Tests are grouped logically.** 15-20 items broken into groups, user works on 1-3 groups at a time.
6. **Skill discovers and documents the dev deployment process.** If a robust, deterministic dev deployment process exists, document it for reuse. If not, help create one.

## Cross-Model Perspective

Codex independently proposed a "persistent test run graph" concept where each node is a test group tied to build/commit hash, environment, deploy recipe, and current status. The key insight: when a fix lands, the system doesn't just re-queue the failed item. It expands the graph, marking affected neighboring checks as invalidated and telling you exactly what became suspect because of the change.

Codex also proposed a concrete file format: `session.yaml` + `groups/*.md` + `deploy.md`, with 6 core commands (init, start, note, fail, fix, resume). The "resume" command is identified as the product.

Closest existing tools: TaskRepo (git-backed markdown state with YAML metadata) gets ~50% there. Mattermost test-management proves repo-native manual test cases can work. The 50% that needs building: test-session semantics, build-aware invalidation, checkpoint integration, deploy recipe discovery, and "what should I test next?" logic.

Both Claude and Codex agree: this is a persistent pair-testing state machine, not a test case manager.

## Approaches Considered

### Approach A: Lightweight Skill (Minimal Viable)
Single skill file that generates a test plan, writes state to a YAML file, tracks pass/fail, and supports resume. No build/deploy integration.
- Effort: S (human: ~2 days / CC: ~20 min)
- Risk: Low
- Pros: Ships fast, immediately useful
- Cons: No rebuild/redeploy loop, no fix invalidation, manual checkpoint discipline

### Approach B: Full Test Session Manager (CHOSEN)
Skill with persistent test session graph. Discovers deploy process, generates grouped test plans from diffs, tracks pass/fail with evidence, auto-checkpoints before fixes, supports resume across sessions. Self-contained state in .context/.
- Effort: M (human: ~1 week / CC: ~45 min)
- Risk: Medium (deploy discovery is project-specific)
- Pros: The complete loop, resume anywhere, builds toward multi-agent orchestration
- Cons: More scope, deploy discovery needs per-project config

### Approach C: Conductor-Native Orchestrator (DEFERRED)
Multi-workspace test sessions. Test plan lives in .context/ and conductor agents can pick up individual test groups. Human reviews results across agents.
- Effort: L (human: ~2 weeks / CC: ~2 hours)
- Risk: High (depends on conductor APIs)
- Pros: Parallel test execution, agent-per-group, scales
- Cons: Over-engineered for now

## Recommended Approach

**Approach B: Full Test Session Manager.** Built with Approach C in mind... the state format and group structure should make multi-agent orchestration a natural extension.

### Skill Name

`/pair-review` — short, memorable, matches the action. Full name: "Pair Testing Session Manager."

### State Format

All state lives in one location:
- `.context/pair-review/` — Conductor-visible, gitignored, self-contained

```
pair-review/
  session.yaml          # Session metadata, active groups, deploy recipe
  deploy.md             # Discovered deploy recipe (how to rebuild/redeploy)
  groups/
    auth.md             # Test group: auth (items, status, evidence, notes)
    onboarding.md       # Test group: onboarding
    dark-mode.md        # Test group: dark mode
```

### Workflow

**Phase 0: Deploy Discovery**

When `/pair-review` is first invoked in a project, the skill looks for an existing deployment process:
1. Check CLAUDE.md for deploy config (`test_deploy_recipe`, `native_app_scheme`, etc.)
2. Check for common patterns: Makefile (dev target), package.json (dev script), conductor-run.sh, docker-compose, etc.
3. If found, write `deploy.md` and confirm with user
4. If not found, explain what's needed and help create one
5. Store the recipe in CLAUDE.md for future sessions

**Phase 1: Test Plan Generation**

Generate the test plan from:
1. `git diff main --stat` — what files changed
2. `git log main..HEAD --oneline` — what commits were made
3. CLAUDE.md, TODOS.md — project context
4. The diff content itself — what specifically changed

Group tests logically (by feature area, screen, or workflow). Each group has 3-7 items. Present the plan to the user for approval/modification via AskUserQuestion.

**Phase 2: Test Execution Loop**

For each active group:
1. Present the next untested item
2. Wait for user result (pass / fail with description / skip / add new item)
3. On PASS: mark passed, move to next
4. On FAIL: record evidence (user description), ask if they want to fix now or continue testing
5. On FIX NOW:
   a. Auto-checkpoint (commit current state)
   b. The agent implements the fix (writes code, not the human)
   c. Rebuild/redeploy using the deploy recipe
   d. Mark the failed item as INVALIDATED (needs retest)
   e. Scan for related items that might be affected by the fix, mark them INVALIDATED
   f. Resume testing
6. On ADD ITEM: add to the current group or create a new group

**Phase 3: Resume**

When `/pair-review resume` is invoked (new session, new machine, or after context compaction):
1. Read session.yaml from ~/.gstack/ (persistent) or .context/ (local)
2. Print a full status dashboard:
   ```
   TEST SESSION: compose-refactor
   Branch: kbitz/compose-refactor | Build: def5678 | Deploy: deploy.md

   AUTH (active):        5/7 passed, 1 failed→fixed, 2 need retest
   ONBOARDING (active):  0/4 tested
   DARK-MODE:            not started (5 items)
   RESPONSIVE:           not started (4 items)

   NEXT: Retest auth items 3, 6, 7 (invalidated by fix def5678)
   DEPLOY: `cd /path/to/bolt && xcodebuild -scheme Bolt ...`
   ```
3. Ask if they want to continue where they left off or start fresh

**Phase 4: Completion**

When all groups are tested and all items pass:
1. Generate a test report (markdown)
2. Save to .context/ for conductor visibility
3. Offer to commit the report
4. Suggest /ship if everything passed

### Integration Points

- **/checkpoint**: The skill auto-checkpoints before every fix attempt using group-level git commits.
- **/ship**: After all tests pass, natural handoff to shipping.
- **/review**: Test results can inform code review (what was manually verified).
- **Conductor .context/**: Other agents can read `.context/pair-review/session.yaml` to understand what's been tested.

### Toward Approach C (Multi-Agent)

The state format is designed so that in the future:
- Each test group could be assigned to a separate conductor agent
- The session.yaml acts as the coordination point
- Groups are independent files, so agents don't conflict
- An append-only audit trail (history.jsonl) could be added for merge-friendly multi-agent coordination

- **Invalidation heuristic:** Dropped for v1. Manual tests are expensive (human time). Over-invalidation is costly. The human decides what to retest. Revisit if real usage shows missed regressions.
- **PR comment:** Deferred to v1.1. Prove the core loop first, then add PR publishing.
- **Checkpoint granularity:** Group-level, not per-item. Commit when starting a test group and when fixing a bug.

## Resolved Issues (from spec review)

### Command Interface (implementer blocker, fixed)

The skill supports these commands:
- `/pair-review` or `/pair-review init` — Start a new test session (deploy discovery + test plan generation)
- `/pair-review resume` — Resume an existing session (status dashboard + continue)
- `/pair-review status` — Print the current dashboard without continuing
- `/pair-review pass [item]` — Mark an item as passed
- `/pair-review fail [item] [description]` — Mark an item as failed with evidence
- `/pair-review skip [item]` — Skip an item (with reason)
- `/pair-review add [group] [description]` — Add a new test item to a group
- `/pair-review fix` — Enter fix mode (auto-checkpoint, implement, rebuild, invalidate)
- `/pair-review done` — Complete the session, generate report

In practice, most of these are invoked conversationally. The user says "that passed" or "this is broken, the loading spinner never stops" and the skill maps it to the right action. The explicit commands exist for resume scenarios where the agent needs to parse intent unambiguously.

### Cross-Machine Survival (structural gap, fixed)

State lives entirely in `.context/pair-review/` (self-contained, no external
dependencies). Cross-machine resume works via:
1. Conductor workspace management (conductor handles workspace state)
2. Committing `.context/pair-review/` to the repo (skill offers this on completion)
No assumptions about Dropbox, iCloud, or any sync service.

### history.jsonl (YAGNI, removed from Approach B)

Removed from the Approach B state format. The append-only audit trail moves to Approach C where it serves as the merge-friendly coordination mechanism for multi-agent test sessions. In Approach B, `session.yaml` and `groups/*.md` fully cover resume and status.

### Deploy Discovery (clarified)

Deploy discovery is best-effort assist, not reliable automation. The primary path is:
1. Check CLAUDE.md for explicit `test_deploy_recipe` config
2. If not found, check common patterns (Makefile dev, npm run dev, etc.)
3. If patterns found, propose and confirm with user
4. If nothing found, ask the user: "What command builds and runs your project for testing?"
5. Always confirm the discovered recipe before first use
6. Store confirmed recipe in CLAUDE.md for future sessions

The user can always override the recipe manually. The skill should never execute an unconfirmed deploy command.

### Auto-Checkpoint (clarified)

"Auto-checkpoint before fix" means:
1. Stage and commit all current changes with message: `test: checkpoint before fix (item {N}: {description})`
2. Record the commit hash in session.yaml checkpoints array
3. If the commit fails (nothing to commit), note "working tree clean" and proceed
4. After the fix, commit the fix with message: `fix: {description} (test item {N})`
5. Record the fix commit hash in the failed item's metadata

This gives a clean revert path. For single-commit fixes: `git revert {fix-commit}`. For multi-commit fixes (fix + follow-up): `git reset --hard {checkpoint-commit}` returns to the pre-fix state. The checkpoint commit hash in session.yaml is the authoritative rollback point.
